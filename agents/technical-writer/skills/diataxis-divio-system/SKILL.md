# Diátaxis (Divio) Documentation System

Diátaxis (formerly Divio's "Documentation System") is the 2026 SOTA framework for organizing docs. Four quadrants, never mix:

|  | Practical | Theoretical |
|---|---|---|
| **Learning** | Tutorial | Explanation |
| **Working** | How-to | Reference |

- **Tutorial** — learning-oriented; the reader is a student; guaranteed success; "Let's build X together."
- **How-to** — task-oriented; the reader has a goal; specific steps; "How to do X."
- **Reference** — information-oriented; the reader needs a lookup; exhaustive; "X is defined as..."
- **Explanation** — understanding-oriented; the reader is curious; context; "Why X works this way."

Mixing them is the #1 cause of unclear docs (a "tutorial" that's actually a reference, or a "reference" full of how-to steps).

## When to use this skill

- Architecting a new docs site from scratch.
- Refactoring an existing site that feels messy.
- Auditing a page that doesn't fit cleanly into any quadrant.

## File-system separation (the structural enforcement)

```
docs/
├── tutorials/          # learning-oriented
│   ├── getting-started.md
│   └── building-your-first-X.md
├── how-to/             # task-oriented
│   ├── deploy-to-aws.md
│   ├── add-authentication.md
│   └── migrate-from-v1.md
├── reference/          # information-oriented
│   ├── api/
│   ├── cli.md
│   ├── configuration.md
│   └── errors.md
└── explanation/        # understanding-oriented
    ├── architecture.md
    ├── why-event-sourcing.md
    └── design-rationale.md
```

This file-system layout becomes the sidebar structure, the URL structure, and (with Vale rules) the linting boundary.

## Vale enforcement of section vocabulary

`.vale.ini` excerpt:

```ini
[docs/tutorials/**/*.md]
BasedOnStyles = Vale, Google, Diataxis
Diataxis.NoHowToHeadings = error

[docs/how-to/**/*.md]
BasedOnStyles = Vale, Google, Diataxis
Diataxis.NoTutorialHeadings = error
Diataxis.MustHaveGoal = error

[docs/reference/**/*.md]
BasedOnStyles = Vale, Google, Diataxis
Diataxis.NoSecondPerson = warning
Diataxis.MustHaveSignature = error

[docs/explanation/**/*.md]
BasedOnStyles = Vale, Google, Diataxis
Diataxis.NoInstructions = warning
```

`.vale/styles/Diataxis/NoTutorialHeadings.yml`:

```yaml
extends: existence
message: 'How-to pages must not contain tutorial framing ("you''ll learn", "in this tutorial"). Move to tutorials/ or rewrite.'
level: error
ignorecase: true
tokens:
  - "(?i)you'll learn"
  - "in this tutorial"
  - "by the end of this lesson"
  - "what you'll build"
```

`.vale/styles/Diataxis/NoHowToHeadings.yml`:

```yaml
extends: existence
message: 'Tutorial pages should not use "How to" headings — those belong in how-to/.'
level: error
scope: heading
ignorecase: true
tokens:
  - '^How to '
```

`.vale/styles/Diataxis/MustHaveGoal.yml`:

```yaml
extends: existence
message: 'How-to pages must state the goal in the first line.'
level: error
scope: paragraph.h1+1
nonword: true
tokens:
  - '(?s)^(?!.{0,100}(How to|This guide|Goal:|Outcome:)).*'
```

`.vale/styles/Diataxis/NoSecondPerson.yml`:

```yaml
extends: existence
message: 'Reference pages are descriptive, not instructional. Avoid "you" — use third person.'
level: warning
ignorecase: true
tokens:
  - '\byou\b'
  - '\byour\b'
  - "\byou're\b"
```

`.vale/styles/Diataxis/MustHaveSignature.yml`:

```yaml
extends: existence
message: 'Reference entries must include a Type / Default / Required / Since block.'
level: error
nonword: true
tokens:
  - '(?s)^\#{1,4}[^\n]+\n(?!(\*\*Type|`|<details))'
```

## Sidebar mapping per generator

### Docusaurus (`sidebars.js`)

```javascript
module.exports = {
  docs: [
    { type: 'category', label: 'Tutorials',   collapsed: false, items: [{type: 'autogenerated', dirName: 'tutorials'}] },
    { type: 'category', label: 'How-to',      collapsed: false, items: [{type: 'autogenerated', dirName: 'how-to'}] },
    { type: 'category', label: 'Reference',   collapsed: true,  items: [{type: 'autogenerated', dirName: 'reference'}] },
    { type: 'category', label: 'Explanation', collapsed: true,  items: [{type: 'autogenerated', dirName: 'explanation'}] },
  ],
};
```

### Starlight (`astro.config.mjs`)

```javascript
sidebar: [
  { label: 'Tutorials',   autogenerate: { directory: 'tutorials' } },
  { label: 'How-to',      autogenerate: { directory: 'how-to' } },
  { label: 'Reference',   autogenerate: { directory: 'reference' } },
  { label: 'Explanation', autogenerate: { directory: 'explanation' } },
],
```

### VitePress (`config.ts`)

```typescript
themeConfig: {
  sidebar: {
    '/tutorials/':   [{ text: 'Tutorials',   items: [...] }],
    '/how-to/':      [{ text: 'How-to',      items: [...] }],
    '/reference/':   [{ text: 'Reference',   items: [...] }],
    '/explanation/': [{ text: 'Explanation', items: [...] }],
  },
}
```

### MkDocs Material (`mkdocs.yml`)

```yaml
nav:
  - Home: index.md
  - Tutorials:   tutorials/
  - How-to:      how-to/
  - Reference:   reference/
  - Explanation: explanation/
```

## Common recipes

### Recipe 1: Refactor existing docs into Diátaxis

1. **Inventory:** list every doc page in a spreadsheet.
2. **Classify:** for each page, mark which quadrant it belongs in (or "mixed" if it spans multiple).
3. **Split mixed pages:** a doc that's half tutorial + half reference becomes two pages.
4. **Move:** apply the new `docs/<quadrant>/` directory structure.
5. **Redirect:** add 301 redirects from old URLs to new (see `docusaurus-vitepress-starlight-mkdocs`).
6. **Vale-gate:** install the Diátaxis style pack; fix all errors before merging.

### Recipe 2: Authoring quadrant cheat sheet

For each page, the agent answers:

- "Is the reader learning a new skill?" → Tutorial.
- "Does the reader have a specific goal in mind already?" → How-to.
- "Is the reader looking up a specific fact?" → Reference.
- "Is the reader trying to understand why?" → Explanation.

If the answer is "two of the above," split the page.

### Recipe 3: Cross-linking between quadrants

The agent links liberally but in the right direction:

- Tutorials link **forward** to how-tos ("Now that you know the basics, try [How to deploy to AWS]").
- How-tos link **down** to reference ("See [API Reference] for all available flags").
- Explanations link **out** to all three ("See [the tutorial] for a hands-on intro and [the reference] for the full API").
- Reference never links elsewhere as a recommendation — it just defines.

### Recipe 4: Title conventions

| Quadrant | Title pattern |
|---|---|
| Tutorial | "Get started", "Build your first X", "Learn X" |
| How-to | "How to <verb>", "Configure X", "Migrate to X" |
| Reference | "X API", "Configuration options", "CLI reference" |
| Explanation | "Why X works this way", "X architecture", "Understanding X" |

## Anti-patterns the agent rejects

- **"Quickstart tutorial that's actually a how-to":** if the reader needs background to follow, it's a how-to.
- **Reference pages with "Let's...":** rewrite as third-person descriptive prose.
- **Explanation pages with step-by-step instructions:** move steps to how-to; leave reasoning.
- **Tutorials with "for production use, see..." disclaimers everywhere:** that's a how-to disguised as a tutorial. Decide and rewrite.

## Edge cases

- **Cookbook recipes:** belong in how-to.
- **API design overview:** belongs in explanation. The API spec is reference.
- **Walkthrough of a single feature:** tutorial if first-time, how-to if user already knows the basics.
- **Glossary / FAQ:** reference if exhaustive lookups; explanation if conceptual.
